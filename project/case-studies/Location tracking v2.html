REAL-TIME TRACKING ENTERPRISE SALES FIELD OPERATIONS

Real-Time Location Intelligence for Field Operations

Built a real-time location tracking system for field operations that unlocked over $25K in enterprise upsell opportunities, delivered 35% efficiency gains for field teams, and became a key differentiator in closing five enterprise deals in Q4. The solution supports the day-to-day operations of 10,000+ managers and technicians.

01 — CONTEXT

Project Overview

Fieldproxy is an AI-powered field service management platform used by operations managers on web and field technicians on mobile to run their entire field operation from a single system. The platform had no reliable way to track and validate technician movement in real time.

I designed the real-time location tracking system from the ground up — a feature that didn't exist in the product before — giving managers live visibility into technician movement, on-site activity, and work hours. This gap had been actively blocking enterprise deals and limiting upsell, a strategic gap we had to close to compete upmarket.

I led the end-to-end design of this real-time location intelligence system, defining the data flows, the mobile and web experiences, and the activity-validation logic.

03 — PROBLEM STATEMENT

What we needed to solve?

Field Operations lacked a reliable real-time system to track and validate technician workforce movement. Reliance on delayed reporting and manual updates led to discrepancies in field activity records, required manual verification, reduced accountability, and weakened trust in operational data.

05 — DESIGN STRATEGY

The Core Design Question

"How do we give enterprise managers real-time field visibility and trust while respecting technician privacy and working within the technical constraints of real-world GPS?"

Every design decision was tested against this question. 

1

Validate the Real Problem

I didn't just assume "location tracking." I interviewed 15 field managers, dispatchers, and technicians across 3 customer sites to understand what decisions actually require real-time visibility.

2

Map Competing Needs

Managers want oversight. Technicians fear surveillance. Engineering is concerned about battery drain and data costs. I had to design a system that satisfied all three stakeholders.

3

Design With Constraints

GPS is ±50m accurate in cities. Battery matters. Data storage is expensive. Rather than pretend these don't exist, I designed the entire experience around them.

Constraints I Designed Around

gps_off

GPS Accuracy ±50m

Urban environments degrade GPS precision. Showing exact coordinates would be misleading — location zones are more honest.

battery_alert

Battery Drain

Continuous GPS polling kills battery. Technicians work 8-10 hour shifts — the system had to be smart about when and how often it pings.

storage

Data Storage Cost

Storing location pings every few seconds across thousands of technicians is expensive. Smart sampling and compression were critical.

06 — DESIGN ITERATION

How the Design Evolved

The first iteration focused on structured activity logs list to improve reporting clarity and task traceability. However, it didn’t surface where activity was happening in real time. Operational decisions required visibility beyond chronological updates, which led to evolving the system into a spatial-first monitoring layer alongside the activity logs.

08 — USER INTERFACE 

The rebuilt platform in action

This location tracking module became the core operational layer for real-time field visibility. As the primary workspace for managers,

Agents overview Screen
This is the primary operational layer for viewing agent field activity. It combines a real-time map with a synchronised activity list to give supervisors a parallel view of movement and work status.

A manager scans the list, and the map dynamically updates the agents. It highlights the relevant technician and their location, enabling the manager to quickly understand where work is taking place without interrupting the agents.

The real-time map displays the active locations of the technicians, while a synchronised activity list reflects each technician's status, including their active time, location, and access. This setup allows for a detailed drill-down into specific information.

The activity card prioritises:

  • Technician identity and work state are surfaced first, followed by time and location signals to validate real-time field activity.
  • Status is dynamically calculated based on activity thresholds, updating when inactivity exceeds defined limits.
  • A drill-down action allows deeper inspection of technician job level details.

Selecting "view" moves from live field tracking into a focused breakdown of the technician's activity view.

Technician Activity

This manager's view provides managers a clear view of what an agent did during the day, including the jobs handled, time spent, and the complete travel path between tasks. It removes guesswork and reduces manual follow-ups by showing activity, travel path, and progress in one structured view.

Field Activity Timeline

  • The timeline view structures in chronological order based on task completion.
  • Each entry records the timestamp, the task name, the status of the task, and the location, creating a traceable account of how actual work unfolded.
  • Aggregated daily metrics — total duration, distance covered, and jobs completed provide a quick field productivity snapshot.
  • A drill-down action reveals task metadata, allowing managers to inspect job-level details.

Field Task Data

  • The task detail consolidates what was assigned, completion timestamp and location, along with technician-submitted response metadata. Together, these elements form a clear and reliable record of the work performed.

Technician Mobile App

Field technicians receive jobs, update statuses, and submit task responses directly from their mobile device — keeping the operational loop tight between the field and the dashboard.

Geofence-Protected Check-In

Technicians can't check in until they arrive at the job location. The geofence (orange tag) enforces arrival verification, preventing false check-ins and reducing fake job claims by anchoring accountability to actual location data.

Radius-Based Location Check

If a technician is outside the job's geofence radius, check-in is blocked. The backend enforces this distance logic, preventing fraudulent check-ins and ensuring work actually happens on-site.

Key Learnings

What did I learn while building real-time tracking?

This project reinforced the importance of designing for data credibility, not just visibility. I learned how real-time systems reduce operational friction by minimizing manual verification and improving decision-making speed.

I also designed for both technical and non-technical users, which required simplifying complex activity data into clear and actionable insights. Close collaboration with cross-functional teams, especially developers, helped me understand system logic and translate it effectively into user experiences.

Moving forward, these insights will guide me in building scalable, data-driven systems that balance technical complexity with usability across diverse user groups.
